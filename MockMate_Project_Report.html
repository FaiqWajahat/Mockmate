1 Project Overview

MockMate is a cutting-edge, AI-driven interview preparation platform that bridges the gap between preparation and performance. It leverages Google Gemini 2.5 Flash Lite to provide real-time, personalized interview practice, ATS-compliant resume analysis, an AI career companion (CoachMate), and a rich admin control centre — all built on a modern full-stack architecture.

The platform follows a five-phase user journey: Preparation → Practice → Analysis → Management (Admin) → Coaching. Each phase is backed by dedicated backend APIs and real-time communication via Socket.io.

🎯 Primary Purpose

Help job-seekers practise interviews with AI feedback, optimise their resumes for ATS systems, and build confidence before real interviews.

🤖 AI at the Core

Every feature — question generation, answer analysis, concept explanations, resume analysis, and CoachMate chat — is powered by Google Gemini.

🔴 Real-Time Features

Socket.io enables a live interview mode with WebRTC-based peer-to-peer video and bidirectional messaging between candidate and interviewer.

3 Core Features

AI Feature 1 — AI Question Generation

What it does: Users specify a job role, years of experience, and topics to focus on. Gemini AI generates a set of tailored interview Q&A pairs (both question and a model answer) in structured JSON format.

How it works: The frontend sends the role/experience/topics to POST /api/ai/generate-questions. The backend builds a prompt using questionAnswerPrompt(), calls Gemini 2.5 Flash Lite, parses and validates the JSON response (with a custom character-level JSON parser to handle malformed output), and returns it to the frontend. Also supports resume-based question generation when a resume is pasted in.

Retry Logic: Implements exponential backoff retry (up to 3 tries, starting at 1s delay doubling each time) for HTTP 429 rate-limit errors.

AI Feature 2 — Live Answer Analysis (Transcript Analysis)

What it does: After a user speaks their answer (recorded via Web Speech API), the transcript is submitted for AI evaluation. Gemini scores the answer (1–10), identifies strengths and areas for improvement, and returns a refined model answer and key takeaways.

How it works: Transcript is sent to POST /api/ai/analyze-transcript. A structured prompt is built with transcriptAnalysisPrompt(question, transcript), the response is parsed to JSON. The frontend renders the score, feedback, and model answer with animations via Framer Motion.

AI Feature 3 — Concept Explanation

What it does: Users can click on any interview question and request a deep AI-generated explanation of the concept — useful for learning unfamiliar topics mid-session.

How it works: Calls POST /api/ai/generate-explanation with the question text. Uses conceptExplainPrompt() to build the prompt and returns a structured explanation object.

AI Feature 4 — ATS Resume Checker

What it does: Users upload their resume (PDF or DOCX) and enter a target job role. Gemini analyses the resume against ATS criteria and returns a score, keyword matches, missing keywords, and improvement suggestions.

How it works: The frontend uses pdfjs-dist (for PDF) and mammoth (for DOCX) to extract resume text on the client side. The extracted text is sent to the backend AI endpoint. The result is displayed as a rich dashboard with charts (Recharts).

AI Feature 5 — CoachMate (AI Career Companion)

What it does: A conversational AI assistant that acts as a friendly "placement bestie". It answers career questions, provides interview tips, links to the user's resume, and guides users to the right MockMate tools — all in a chat interface.

How it works: Uses a detailed system prompt that defines CoachMate's personality, routing map, and response style. The last 10 messages of conversation history are included in each prompt for context. Chat history is persisted in MongoDB via the CoachMateChat model. Supports history retrieval and resets.

Real-Time Feature 6 — Live Interview Mode (WebRTC + Socket.io)

What it does: A full peer-to-peer video interview simulation. One user acts as the interviewer, another as the candidate. Both join a shared room, see each other via live video, and can chat simultaneously.

How it works: Socket.io handles signalling: join-room notifies peers, signal relays WebRTC offers/answers/ICE candidates between peers. SimplePeer (a WebRTC library) handles the direct video stream. The backend does not process video — it only acts as the signalling server.

Sessions Feature 7 — Session-Based Interview Prep

What it does: Users create a named "prep session" tied to a specific role and set of topics. Within a session, Gemini generates questions that are saved to the database. Users can pin important questions, add personal notes, and revisit sessions from their Dashboard.

How it works: The Session model stores the user reference, role, experience, topics, and linked question IDs. The Question model stores each Q&A pair with note and isPinned flag. The Dashboard displays all sessions with timestamps.

Auth Feature 8 — Authentication (Email/Password + Google OAuth)

What it does: Users can sign up and log in with email/password OR sign in instantly with Google. All protected routes require a valid JWT token in the Authorization header.

How it works: For email auth, passwords are hashed with bcryptjs (salt rounds: 10) and JWTs are signed with JWT_SECRET (7-day expiry). For Google auth, Firebase handles the OAuth flow on the frontend; the resulting user profile (name, email, photo) is sent to POST /api/auth/google, which upserts the user in MongoDB and returns a JWT. The protect middleware verifies JWTs on all private endpoints.

Admin Feature 9 — Admin Control Panel

What it does: Admins have a dedicated control panel to view all registered users, delete users (cascades to their sessions and questions), manage toast notifications shown app-wide, and send broadcast messages/emails to all users or selected users.

How it works: Admin routes under /api/admin. The Broadcast model records all sent messages. The Toast model allows admins to create sequenced, delayed toast notifications that appear globally for all users. The Notification model tracks per-user read/unread status.

PDF Feature 10 — Interview Report PDF Generation

What it does: After an analysis, users can generate and download a PDF report containing the question, their transcript, AI score with grade (A+–D), strengths, improvements, refined answer, and performance metrics.

How it works: The backend endpoint POST /api/ai/generate-pdf-data structures the raw analysis into a rich data object with metadata, interview details, and performance metrics. The frontend then uses jsPDF + html2canvas to render and download the PDF in-browser.

Resume Feature 11 — Bento Resume Workspace

What it does: A bento-grid style workspace where users can store their resume link, view it inline, get AI-powered improvement tips, and quickly navigate to ATS analysis. Resume links are persisted to the user profile.

How it works: The PUT /api/auth/resume-link endpoint updates the resumeLink field on the User document. CoachMate also reads this link to answer questions about the user's resume.

Practice Feature 12 — HR Interview Practice (Voice + AI)

What it does: A dedicated mode for behavioural / HR round practice. Users answer questions by speaking; the browser's Web Speech API captures and transcribes their answer in real time; Gemini then analyses the transcript.

How it works: Frontend uses the window.SpeechRecognition Web API. After recording, the transcript is cleaned via POST /api/ai/cleanup-transcript (removes filler words and fixes grammar) before being sent for full analysis at POST /api/ai/analyze-transcript.

7 Real-Time Communication (Socket.io)

The backend creates a single Socket.io server instance attached to the same HTTP server as Express. All real-time events flow through this single connection point.
Event (Client → Server) Direction Purpose
join_room Client → Server Join a named chat room
send_message Client → Server Send a chat message to a room
receive_message Server → Client Relay message to other room members
join-room Client → Server Join a WebRTC video room
user-joined Server → Client Notify peers a new user joined
signal Bidirectional Relay WebRTC offer / answer / ICE candidates
other-user Server → Client Send WebRTC offer with peer ID to initiator
disconnect Server event Log user disconnection

WebRTC Video Flow

  • User A visits /interview/live and emits join-room with a room ID.
  • User B joins the same room — server emits user-joined to User A.
  • User A creates a WebRTC offer via SimplePeer and emits signal (type: offer).
  • Server relays the offer to User B as other-user.
  • User B sends back an answer via signal; ICE candidates flow both directions.
  • Peer-to-peer video stream established — no video data passes through the server.

8 Security Architecture

🔑 JWT Authentication

All private endpoints validate a Bearer token in the Authorization header. Tokens are signed with JWT_SECRET and expire in 7 days.

🔒 Password Hashing

bcryptjs with salt rounds of 10 is used to hash passwords at registration. Passwords are never returned in API responses.

🌐 CORS Policy

Server explicitly allows only known origins (localhost ports, Vercel, Netlify, Render domains) for both REST and Socket.io connections.

🛡️ OAuth via Firebase

Google login uses Firebase OAuth on the frontend. Only the verified profile info is sent to the backend — no Google tokens are exposed to the backend server.

🚫 Cascade Delete

When an admin deletes a user, all associated Sessions and Questions are deleted first, preventing orphaned database records.

📋 Error Logging

AI API errors and rate-limit events are appended to error_log.txt server-side with ISO timestamps for debugging.
